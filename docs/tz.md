# Техническое задание: Dating-приложение (Telegram Bot + Monolithic Backend)

## 1. Описание проекта
Проект представляет собой dating-приложение в формате Telegram-бота. Пользователь взаимодействует с системой через Telegram-бот:
- регистрируется
- заполняет анкету
- загружает фотографии
- просматривает анкеты других пользователей
- ставит like / skip
- получает match при взаимном лайке
- получает уведомления о лайках и мэтчах
- отправляет жалобы

Backend реализуется как монолитное приложение с модульной внутренней структурой.

---

## 2. Архитектура системы

### Общая схема
`Telegram User` → `Telegram Bot` → `FastAPI Backend` → `PostgreSQL`

**Дополнительные инфраструктурные компоненты:**
- `FastAPI` → `Redis`
- `FastAPI` → `RabbitMQ`
- `RabbitMQ` → `Celery Workers`
- `FastAPI` → `MinIO`

### Архитектурный принцип
Используется модульный монолит:
- один backend
- одна база данных
- отдельные бизнес-модули внутри проекта
- асинхронная обработка тяжёлых операций через очередь

---

## 3. Технологии

### FastAPI
Основной backend API. Используется для:
- REST API
- бизнес-логики
- работы с БД
- интеграции с Redis / RabbitMQ / MinIO

### Telegram Bot API
Пользовательский интерфейс системы. Через бота выполняются все пользовательские действия.

### PostgreSQL
Основное хранилище данных. Используется для:
- пользователей и анкет
- фотографий
- свайпов и мэтчей
- рейтингов и жалоб

### Redis
Используется для кэширования feed анкет.
- **Задача:** Предварительно сохранять список следующих анкет для пользователя.
- **Ключ:** `feed:user:{id}`
- **Хранится:** список `profile_id`

### RabbitMQ
Используется как очередь событий внутри монолита.
Через RabbitMQ отправляются события:
- `like_created` → триггер для создания уведомления
- `match_created` → триггер для отправки уведомления обоим пользователям
- `complaint_created`
- `photo_uploaded`
- `notification_send` → отправка уведомления в Telegram
- **Задача:** Не выполнять тяжёлую бизнес-логику синхронно в API.

### Celery
Используется для фоновых задач. Выполняет:
- перерасчёт рейтинга
- предварительное обновление feed
- отправку уведомлений в Telegram

### MinIO
Используется для хранения фотографий анкет.
- **В БД хранится:** только ссылка на файл.

## 4. Модули backend

### `users` (Основной модуль пользователя и анкеты)
Отвечает за:
- регистрацию пользователя и создание/редактирование анкеты
- хранение предпочтений поиска и города
- выбор режима поиска (только свой город / все города)
- загрузку фотографий

### `swipes` (Модуль взаимодействия с анкетами)
Отвечает за:
- like / skip и историю просмотров
- *После like:* отправляет событие в RabbitMQ.

### `matches` (Модуль мэтчей)
Отвечает за:
- проверку взаимного лайка и создание мэтча
- фиксацию начала диалога
- *После match:* отправляет событие в RabbitMQ.

### `ratings` (Модуль рейтинга анкет)
Отвечает за:
- первичный рейтинг анкеты
- накопительный score и начисление баллов за активность
- штрафы за жалобы и снижение рейтинга за неактивность

### `tasks` (Фоновые задачи)
Выполняет через Celery:
- перерасчёт рейтинга и обработку событий RabbitMQ
- обновление Redis feed и daily decay

### `core` (Инфраструктурный слой)
Содержит конфигурацию и подключения к PostgreSQL, Redis, RabbitMQ, MinIO.

## 5. Структура базы данных

- **users:** `id`, `telegram_id`, `created_at`, `last_active_at`, `referral_code`, `invited_by`, `complaints_count`, `status`
- **profiles:** `id`, `user_id`, `name`, `age`, `gender`, `city`, `bio`, `interests`, `preferred_gender`, `preferred_age_min`, `preferred_age_max`, `search_city_mode` (`local` / `global`)
- **photos:** `id`, `profile_id`, `photo_url`, `position`
- **swipes:** `id`, `from_user_id`, `to_profile_id`, `action`, `created_at`
- **matches:** `id`, `user1_id`, `user2_id`, `created_at`, `dialog_started`
- **ratings:** `user_id`, `base_rank`, `updated_at`
- **rating_history:** `id`, `user_id`, `event_type`, `delta`, `created_at`
- **complaints:** `id`, `from_user_id`, `to_user_id`, `reason`, `created_at`

## 6. Рейтинг анкет

### Первичный рейтинг
Начисляется после заполнения анкеты. Учитывается:
- заполненность анкеты
- наличие фото
- наличие интересов и предпочтений

### Base Rank (MVP)
На этапе MVP рейтинг анкеты хранится в поле `base_rank` и считается из трех нормализованных сигналов `[0..1]`:
- заполненность фото (`photo_count_score`) с весом `0.2`
- длина био (`bio_length_score`) с весом `0.2`
- активность (`activity_days_score`) с весом `0.6`

Итоговая формула:
`base_rank = (photo_count_score * 0.2) + (bio_length_score * 0.2) + (activity_days_score * 0.6)`

---

## 7. Алгоритм выдачи анкет

1. **SQL фильтрация.** Исключаются: собственная анкета, просмотренные анкеты, неподходящий возраст, неподходящий пол, неактивные профили.
2. **Географический фильтр.** Если `local` — только анкеты своего города. Если `global` — город не учитывается.
3. **Сортировка.** По `base_rank` (`ORDER BY base_rank DESC`) или по результату ranking service.
4. **Redis.** После первой выдачи сохраняются следующие 10 анкет.

## 8. Асинхронная обработка

**RabbitMQ события:**
- `like_created`, `match_created`, `complaint_created`, `photo_uploaded`

**Celery consumers:**
- Получают события и выполняют: изменение рейтинга, создание match, запись истории рейтинга, обновление feed.

## 9. Основной пользовательский flow
`register` → `create profile` → `upload photos` → `get feed` → `like / skip` → `match` → `rating update` → `feed preload`

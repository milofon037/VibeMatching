# ПРАКТИКА: СРАВНЕНИЕ RABBITMQ И REDIS КАК БРОКЕРОВ СООБЩЕНИЙ
## ЦЕЛЬ

Сравнить `RabbitMQ` и `Redis` в одинаковых условиях и получить измеримый результат:

- пропускную способность;
- влияние размера сообщения;
- момент, когда `single instance` начинает деградировать или падать.

## ФОРМАТ

Стенд из 3 частей:

- `producer` -> генерирует и отправляет сообщения;
- `broker` -> по очереди `RabbitMQ` и `Redis`;
- `consumer` -> читает и считает успешно обработанные сообщения.

Важно: для обеих технологий условия должны быть одинаковыми:

- один и тот же формат сообщений;
- одинаковое число producers и consumers;
- одинаковая длительность тестов;
- одинаковые лимиты по ресурсам.

## ГЛАВНЫЙ УПОР В ЭТОЙ ПРАКТИКЕ

Нужно не просто подключить брокеры, а провести серию прогонов и сравнить результат.

Обязательные параметры сравнения:

- размер сообщения, например: `128 B`, `1 KB`, `10 KB`, `100 KB`;
- интенсивность потока, например: `1 000 msg/sec`, `5 000 msg/sec`, `10 000 msg/sec`;
- общий объем сообщений или длительность теста.

## ОБЯЗАТЕЛЬНЫЕ ЭКСПЕРИМЕНТЫ

### 1. БАЗОВОЕ СРАВНЕНИЕ

Прогнать одинаковую нагрузку через:

- `RabbitMQ`;
- `Redis`.

Сравнить:

- сколько сообщений отправлено;
- сколько сообщений обработано;
- число ошибок.

### 2. ВЛИЯНИЕ РАЗМЕРА СООБЩЕНИЯ

Проверить, как меняется производительность при увеличении payload.

Нужно ответить:

- какой брокер лучше работает с маленькими сообщениями;
- какой быстрее деградирует на больших сообщениях.

### 3. ВЛИЯНИЕ ИНТЕНСИВНОСТИ ПОТОКА

Постепенно увеличивать нагрузку, пока `single instance` не начнет:

- накапливать очередь;
- заметно увеличивать latency;
- выдавать ошибки;
- переставать справляться с потоком.

Нужно зафиксировать, при какой нагрузке это произошло для каждой технологии.

## ИНСТРУМЕНТЫ

Можно использовать:

- собственный `producer`;
- `k6`;
- `Locust`;
- `JMeter`;
- любой другой инструмент для stress/load testing.

## ЧТО ИЗМЕРИТЬ

Минимум нужно собрать:

- `messages/sec`;
- среднюю задержку;
- максимальную задержку или `p95`;
- количество успешно обработанных сообщений;
- количество потерянных сообщений;
- точку, где `single instance` перестает справляться.

Дополнительно, если получится:

- CPU;
- RAM;
- backlog / размер очереди.

## ЧТО СДАТЬ

- Код и скрипты запуска тестов;
- Отчет со скринами, вашими выводами и таблицей результатов.

## ЧТО ДОЛЖНО БЫТЬ В ВЫВОДАХ

- какой брокер показал большую пропускную способность;
- какой брокер лучше переносит увеличение размера сообщения;
- при какой нагрузке `single instance RabbitMQ` и `single instance Redis` начинают деградировать;
- какой инструмент лучше подходит для такого сценария и почему.

# РЕШЕНИЕ
## Полноценное решение по ТЗ: RabbitMQ vs Redis

Реализация выполнена в папке `pr_brokers` и включает код, автоматизацию прогонов и генерацию отчета.

## 1. Что реализовано

- стенд в Docker Compose с ограничением ресурсов для `single instance`;
- собственный producer/consumer на Python для двух брокеров;
- подтверждения доставки и ACK в обоих вариантах:
  - RabbitMQ: `publisher confirms` + `basic_ack`;
  - Redis Streams: `XADD` + `XREADGROUP` + `XACK`;
- матрица экспериментов по размерам сообщений и интенсивности;
- отдельные stress-прогоны для поиска точки деградации;
- агрегатор результатов в `CSV` и `Markdown` отчет.

## 2. Структура файлов решения

- `docker-compose.yml` - RabbitMQ + Redis с лимитами CPU/RAM;
- `benchmark.py` - один прогон (один брокер, один набор параметров);
- `run_experiments.sh` - полный автоматический запуск серии тестов;
- `aggregate_results.py` - сводный отчет (`results.csv` + `report.md`);
- `README.md` - инструкция по запуску и интерпретации метрик;
- `requirements.txt` - зависимости Python.

## 3. Методология тестирования

- одинаковые условия для RabbitMQ и Redis;
- warm-up перед измерениями;
- payload нужного размера с рандомной строкой;
- фиксация метрик по latency/throughput/loss/backlog;
- критерий деградации:
  - обработано < 95% отправленных, или
  - p95 > 500 ms, или
  - backlog > 10% от отправленных.

## 4. Набор экспериментов

Базовая матрица:

- payload: `128 B`, `1 KB`, `10 KB`, `100 KB`;
- target RPS: `1000`, `5000`, `10000`;
- для каждого набора запускаются оба брокера.

Stress-прогоны:

- для `1 KB`: RPS `15000`, `20000`, `30000`, `40000`, `50000`;
- для `100 KB`: RPS `1000`, `3000`, `5000`.

## 5. Как запускать

```bash
cd VibeMatching/practices/pr_brokers
chmod +x run_experiments.sh
./run_experiments.sh
```

Результаты сохраняются в `results/<timestamp>/`:

- `raw.ndjson` - сырые записи по каждому прогону;
- `results.csv` - удобная таблица для анализа;
- `report.md` - итоговая сводка с победителем по ключевым метрикам.

## 6. Что сдавать по практике

1. Код и скрипты из `pr_brokers`.
2. Папку `results/<timestamp>/` с фактическими измерениями.
3. Скрины Docker/графиков и выводы по вопросам ТЗ:
   - кто дал выше throughput;
   - кто лучше переносит рост payload;
   - при каком RPS началась деградация для каждого брокера.

## 7. Ожидаемая интерпретация результатов

- Redis обычно показывает более низкую latency на малых payload;
- RabbitMQ чаще устойчивее по сценариям очередей, но с большим protocol overhead;
- на больших payload оба деградируют, но характер деградации различается:
  - RabbitMQ: рост backlog и latency при насыщении CPU/IO;
  - Redis: упор в RAM и ограничения single-thread execution.

Фактический победитель определяется только по вашим измерениям в `report.md`.

## 5. Сравнение по всем критериям

### 5.1 Пропускная способность

На `target_rps=5000`:
- `128 B`, `1 KB`: выигрывает Redis.
- `10 KB`: Redis немного выше.
- `100 KB`: RabbitMQ немного выше.

По максимуму throughput в стресс-сценариях:
- `128 B`: RabbitMQ `3268.9`, Redis `4682.53` -> Redis лучше.
- `1 KB`: RabbitMQ `2357.55`, Redis `3420.85` -> Redis лучше.
- `10 KB`: RabbitMQ `758.83`, Redis `765.87` -> Redis немного лучше.
- `100 KB`: RabbitMQ `105.05`, Redis `102.25` -> RabbitMQ немного лучше.

### 5.2 Задержки

- По `p95` Redis лучше на всех payload.
- На `100 KB` у RabbitMQ наблюдается резкий рост хвоста задержек (`p95` ~ `11.5s`), тогда как у Redis `p95` ~ `30ms`.
- Средняя задержка подтверждает ту же картину: на `100 KB` у RabbitMQ `avg` ~ `7091ms`, у Redis `avg` ~ `20.7ms`.

### 5.3 Потери и backlog

- Потерянные сообщения: `0` для обоих брокеров во всех ключевых сценариях.
- Медианный backlog в итоговых сценариях: `0`.

### 5.4 CPU и RAM

- На малых payload RabbitMQ заметно более CPU-интенсивен, Redis экономичнее по CPU.
- На `10 KB` Redis использует больше RAM, но держит более низкий p95.
- На `100 KB` RAM у брокеров сопоставима, но поведение по latency принципиально разное (Redis существенно стабильнее).

## 9. Почему Redis хуже на больших данных (и в чем именно хуже)

Важно: в текущем прогоне Redis хуже на больших payload в основном по **throughput** (не по latency).

Что видно по данным:
- На `100 KB` throughput у RabbitMQ немного выше (`103.70` vs `100.20` msg/s на срезе `5000 RPS`).
- При этом по задержкам Redis значительно лучше (`p95 30.184 ms` vs `11455.446 ms`).

Почему так происходит:
1. Redis Streams обрабатываются в одном event-loop потоке Redis.
  На крупных payload растет стоимость одной операции `XADD/XREADGROUP/XACK` по копированию и обслуживанию структуры stream.
2. Для больших сообщений увеличивается накладной расход на управление памятью и внутренние структуры Redis (entries, индексы, bookkeeping consumer group).
3. Redis настроен с `maxmemory` и `noeviction`.
  При приближении к лимитам он не вытесняет данные, а снижает способность принимать дополнительную запись быстро.
4. RabbitMQ в этой конфигурации может чуть лучше утилизировать конвейер доставки крупных transient-сообщений по throughput,
  но платит за это тяжелым хвостом задержек на больших payload.

Итог по интерпретации:
- Если критерий приоритета — максимальный throughput на очень крупном payload, RabbitMQ в этом прогоне чуть впереди.
- Если критерий приоритета — предсказуемые задержки и стабильный p95, Redis на больших payload заметно лучше.

## 10. Финальный вывод по прогону `20260421_230330`

1. Redis является общим победителем по совокупности метрик для `128 B`, `1 KB`, `10 KB`.
2. На `100 KB` RabbitMQ немного лучше по throughput, но резко хуже по latency.
3. Устойчивая деградация зафиксирована у RabbitMQ на `100 KB @ 1000 RPS` (`6/6` деградировавших повторов).
4. Для Redis в протестированном диапазоне устойчивая деградация не достигнута.
5. Для продакшн-выбора при интерактивных нагрузках и SLA по latency результат этого прогона явно в пользу Redis.

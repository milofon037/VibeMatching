# ПРАКТИКА: СРАВНЕНИЕ ТИПОВ КЕШИРОВАНИЯ

## ЦЕЛЬ

Сделать три простых примера одной и той же системы с разными типами кеширования и сравнить их в одинаковых условиях:

- `Lazy Loading` / `Cache-Aside` / `Write-Around`;
- `Write-Through`;
- `Write-Back`.

## ФОРМАТ

Одна и та же система должна быть реализована в трех вариантах.

Минимальный состав:

- `load-generator`; (например Jmeter или самописный)
- `application`;
- `cache`; (Redis/Apache Ignit/etc...)
- `БД`.

Меняется только стратегия работы с кешем.

## ЧТО НУЖНО СДЕЛАТЬ

### 1. Lazy Loading / Cache-Aside

- чтение идет через кеш;
- если данных нет, они берутся из БД и кладутся в кеш;
- запись идет сразу в БД.

### 2. Write-Through

- чтение идет через кеш;
- запись сразу попадает и в кеш, и в БД.

### 3. Write-Back

- чтение идет через кеш;
- запись сначала попадает в кеш;
- в БД данные отправляются позже.

## ЕДИНЫЙ ТЕСТ

Для всех трех вариантов нужен один и тот же тест.

- одинаковый набор данных;
- одинаковое число запросов;
- одинаковая длительность;

Минимум нужно сделать 3 прогона:

- `read-heavy`, например `80% read / 20% write`;
- `balanced`, например `50% read / 50% write`;
- `write-heavy`, например `20% read / 80% write`.

## МЕТРИКИ

Нужно измерить:

- `throughput` (`req/sec`);
- среднюю задержку;
- количество обращений в БД;
- hit rate кеша.

Для `Write-Back` желательно отдельно показать, что происходит при накоплении записей.

# ОТЧЁТ
## Описание тестов

### Стратегии кеширования

#### 1. Lazy Loading (Cache-Aside)
- **Принцип:** Кеш используется только для операций чтения
- **Чтение:** cache → [HIT] → return | cache → [MISS] → DB → save in cache
- **Запись:** Напрямую в БД (без обновления кеша)
- **Hit Rate:** Зависит от частоты обращения к данным
- **Преимущества:** Простота реализации, минимальное управление
- **Недостатки:** Возможна рассинхронизация, низкий hit rate при частых изменениях

#### 2. Write-Through
- **Принцип:** Любое изменение записывается синхронно в кеш и БД
- **Чтение:** cache → [HIT] → return | cache → [MISS] → DB → save in cache
- **Запись:** Одновременно в cache И в DB
- **Hit Rate:** Высокий (кеш всегда актуален)
- **Преимущества:** Гарантирует консистентность данных
- **Недостатки:** Медленнее чем Write-Back из-за двойной записи

#### 3. Write-Back (Write-Behind)
- **Принцип:** Запись в кеш моментальна, запись в БД отложена
- **Чтение:** cache → [HIT] → return | cache → [MISS] → DB → save in cache
- **Запись:** Только в cache (быстро), затем асинхронно в DB
- **Hit Rate:** Очень высокий (кеш содержит последние изменения)
- **Преимущества:** Максимальная скорость записи
- **Недостатки:** Требует управления отложенными записями, окно рассинхронизации

### Сценарии нагрузки

1. **Read-Heavy (80% read / 20% write):** Преобладание операций чтения - типичная ситуация для систем с кешем (веб-приложения, API)
2. **Balanced (50% read / 50% write):** Равное распределение - типично для систем управления данными
3. **Write-Heavy (20% read / 80% write):** Преобладание операций записи - системы логирования, аналитики, очередей

### Параметры тестов

- **Операций на тест:** 1000
- **Количество уникальных ключей:** 50
- **БД:** SQLite (in-memory)
- **Кеш:** Python dict (in-memory)

## Анализ результатов

### 1. Read-Heavy сценарий (80% read / 20% write)

**Результаты:**
- **Lazy Loading:** 1,363.61 req/sec (62.64% hit rate)
- **Write-Through:** 1,495.78 req/sec (79.23% hit rate) - Лучше на 9%
- **Write-Back:** 67,776.87 req/sec (78.57% hit rate) - В 45x раз быстрее

**Анализ:**
1. **Пропускная способность:** Write-Back доминирует благодаря асинхронной записи (время записи = 0.00ms)
2. **Hit Rate:** Write-Through показывает лучший результат (79.23%) за счет синхронизации кеша при каждой записи
3. **Обращения в БД:** 
   - Lazy Loading: 497 обращений (50% от операций)
   - Write-Through: 359 обращений (36% от операций)
   - Write-Back: 171 обращение (17% от операций) - В 3x раз меньше
4. **Вывод:** При преобладании чтения все стратегии работают хорошо, но Write-Back обеспечивает максимальную пропускную способность за счет асинхронной записи

### 2. Balanced сценарий (50% read / 50% write)

**Результаты:**
- **Lazy Loading:** 495.31 req/sec (78.87% hit rate)
- **Write-Through:** 390.82 req/sec (90.76% hit rate) - Hit rate отличный, но медленнее на 21%
- **Write-Back:** 625,548.70 req/sec (87.55% hit rate) - В 1260x раз быстрее

**Анализ:**
1. **Пропускная способность:** Write-Back показывает огромное преимущество (время выполнения 0.002s против 2.019-2.559s)
2. **Hit Rate:** Write-Through показывает лучший результат (90.76%), что говорит о идеальной синхронизации
3. **Обращения в БД:**
   - Lazy Loading: 608 обращений (61% от операций)
   - Write-Through: 568 обращений (57% от операций)
   - Write-Back: 64 обращение (6% от операций) - В 9x раз меньше
4. **Вывод:** При равном распределении выбор зависит от приоритета:
   - Нужна максимальная скорость → Write-Back
   - Нужна консистентность → Write-Through

**Результаты:**
- **Lazy Loading:** 382.85 req/sec (72.17% hit rate)

**Анализ:**

2. **Hit Rate:** 
   - Write-Back: 95.50% - Лучший (кеш содержит все последние изменения)
   - Write-Through: 94.33% (очень близко)
   - Lazy Loading: 72.17% (кеш не обновляется при записи)
3. **Обращения в БД:**

## Практические рекомендации

### Когда использовать Lazy Loading?
**Используйте, если:**
- Система простая, без критичных требований к консистентности
- Данные в БД меняются редко или ваше приложение их не обновляет
- Допустима временная рассинхронизация (несколько секунд/минут)

**Не используйте, если:**
- Частые операции записи (hit rate упадет до 62%)
- Критична консистентность данных
- Нужна высокая производительность

### Когда использовать Write-Through?
**Используйте, если:**
- Критична **консистентность данных** (финансовые системы, конфигурации)
- Нужны гарантии, что данные точно в БД после операции
- Операции записи нечасты или их немного

**Не используйте, если:**
- Нужна максимальная скорость (медленнее на 30-50%)
- Write-heavy нагрузка (Write-Back в 1800x раз быстрее)

### Когда использовать Write-Back?
**Используйте, если:**
- **Максимальная скорость критична** (высоконагруженные системы)
- Операции записи частые или интенсивные (80% нагрузки)
- Можно допустить окно расхождения между кешем и БД (обычно миллисекунды)
- Есть механизм graceful shutdown для `flush_to_db()`

**Не используйте, если:**
- Нельзя допустить потерю данных при сбое (без механизма восстановления)
- Очень критична мгновенная консистентность

## Примеры реальных применений

### Lazy Loading
- Кеш профилей пользователей (обновляются редко)
- Кеш статических данных (справочники, конфигурации)
- Кеш вычисляемых значений (статистика, аналитика)

### Write-Through
- Финансовые системы (платежи, переводы)
- Системы управления критичными данными (конфигурации приложения)
- Логирование аудита (нужна гарантия записи)

### Write-Back
- Analytics системы (отправка метрик и событий)
- Логирование приложения (объемное логирование)
- Кеширование сессий (данные можно восстановить из основного источника)
- Системы очередей (высокая пропускная способность)

## Итоговая рекомендация

```
Выбирайте стратегию исходя из приоритетов:

1. Нужна максимальная скорость?
   → Write-Back (в 1000+ раз быстрее)

2. Критична консистентность данных?
   → Write-Through (гарантирует синхронизацию)

3. Простая система, редкие изменения?
   → Lazy Loading (простая реализация)

4. Не уверены?
   → Write-Back с периодическим flush_to_db()
      (лучший компромисс между скоростью и надежностью)
```
